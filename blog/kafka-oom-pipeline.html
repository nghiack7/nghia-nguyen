How I Fixed a 100M+ Message Pipeline Bottleneck and Recurring OOM Crashes

Lessons from debugging unbounded concurrency, memory leaks, and backpressure in a high-throughput Go event processor

NN
Nghia Nguyen
Senior Software Engineer II  ·  TrueProfit  ·  April 2026
~12 min read

At TrueProfit, we process real-time e-commerce analytics for Shopify merchants — tracking ads, revenue, and attribution across hundreds of thousands of events per second. At the heart of this is TruePixel, our event processing pipeline.

Over the past year, I tackled two critical production incidents that were causing hundreds of millions of messages to back up in Kafka and recurring OOM kills on our event processor pods. This post covers the root causes, the fixes, and the hard lessons learned.

The Architecture

The Event Processor consumes from Kafka, processes each event through 5 stages, and writes to Elasticsearch, MongoDB, and Redis. Simple enough — until it isn't.

Incident 1: The 100M+ Message Bottleneck

🚨
Production Impact
Kafka consumer lag growing unbounded. Hundreds of millions of messages stuck. Pods restarting under load.

What Went Wrong

The original OrderedEventProcessor (1,291 lines, now deleted) had a critical design flaw: unbounded per-user goroutines.

ordered_event_processor.go (BROKEN) 
// BROKEN: Creates a new goroutine for EVERY unique user
// sync.Map grows without limit
type OrderedEventProcessor struct {
    userQueues sync.Map  // key: "shopID:clientID" → goroutine
    // No limit on how many goroutines can exist
}

For every unique shopID:clientID pair, a new goroutine was spawned with its own processing queue. With 50,000+ unique users hitting the system, that's 50,000+ goroutines — each holding memory, each competing for CPU.

The Fix: Bounded Keyed Dispatcher

I replaced the entire OrderedEventProcessor with a bounded userDispatcher — the key innovation that solved everything.

The dispatcher routes messages by user key to a bounded worker pool. When the queue is full (512 messages or 32 MiB), it blocks the Kafka consumer loop — proper backpressure instead of infinite buffering.

user_dispatcher.go 
// The key insight: block Kafka when we can't keep up
func (d *userDispatcher) dispatch(msg *sarama.ConsumerMessage) {
    for {
        d.mu.Lock()
        if d.pendingMessages >= d.maxPendingMessages ||
           d.pendingBytes+msgSize > d.maxPendingBytes {
            d.mu.Unlock()
            <-d.budgetAvailable  // BLOCK until a worker drains
            continue
        }
        d.queues[userKey] = append(d.queues[userKey], msg)
        d.pendingMessages++
        d.pendingBytes += msgSize
        d.mu.Unlock()
        break
    }
}

For product stats, I switched from "load everything into memory" to batch processing:

product_stats_flusher.go 
// BEFORE: Load all 50K Redis keys at once → 480 MiB spike
keys := getAllKeysFromRedis()  // 50,000 keys in memory

// AFTER: Process in batches of 100 → ~10 MiB spike
for batch := range scanKeysInBatches(100) {
    bulkWriteToES(ctx, batch)
}

And pre-computed the Lua script SHA at startup instead of calling SCRIPT LOAD 500 times/sec:

redis_scripts.go 
// BEFORE: 500 SCRIPT LOAD calls/second
redis.Call("EVAL", luaScript, ...)

// AFTER: Pre-compute SHA once at startup
var sha = computeSHA(luaScript)
redis.Call("EVALSHA", sha, ...)  // Zero extra round-trips

Incident 2: The Recurring OOM Crashes

💥
Production Impact
Even after fixing the bottleneck, OOM kills kept happening — but for a different reason. Peak memory 1.2–1.5 GB, pods killed 2–3x/week.

Root Cause: Unbounded Cache + Context Timeout Leaks

The visitor tracking logic used a Ristretto in-memory cache configured with absurd limits:

visitor_cache.go (BROKEN) 
// BROKEN: 1 GB cache with 10M key slots
cache, _ := ristretto.NewCache(&ristretto.Config{
    NumCounters: 1e7,      // 10 million keys
    MaxCost:     1 << 30,  // 1 GB budget
})

Combined with a single 60-second timeout wrapping cache + Elasticsearch + MongoDB operations:

visitor_processor.go (BROKEN) 
// BROKEN: One timeout for everything
ctx, cancel := context.WithTimeout(baseCtx, 60*time.Second)
defer cancel()

checkCache(ctx)            // <1ms
queryElasticsearch(ctx)    // could wait 60s
writeMongoDB(ctx)          // could wait 60s

If ES was slow, the entire chain waited. And Info-level logs in this hot path (1,000+ visitors/sec = 1,000+ log allocations/sec) added serious GC pressure.

The Fix

visitor_processor.go (FIXED) 
// FIX 1: Halve the cache budget
cache, _ := ristretto.NewCache(&ristretto.Config{
    NumCounters: 5e6,       // 5M keys (was 10M)
    MaxCost:     500 << 20, // 500 MB (was 1 GB)
})

// FIX 2: Per-operation timeouts
esCtx, esCancel := context.WithTimeout(baseCtx, 30*time.Second)
existsInES, _ := repo.VisitorExistsInES(esCtx, ...)
esCancel()  // Release immediately

mongoCtx, mongoCancel := context.WithTimeout(baseCtx, 15*time.Second)
repo.IncrementWeeklyUniqueVisitors(mongoCtx, ...)
mongoCancel()

// FIX 3: Debug logs instead of Info in hot path
logger.Debug("visitor cache hit")  // was Info

Impact: Peak memory dropped from 1.2–1.5 GB to 400–500 MB. OOM kills: eliminated.

Key Takeaways

01
Unbounded concurrency is a time bomb

A sync.Map that spawns goroutines "looks safe" but creates a leak that only manifests under sustained load. Always set explicit bounds on any concurrent resource.

02
Backpressure > buffering

Without backpressure, Kafka keeps pushing messages into an overwhelmed consumer. The fix: block the consumer loop when the memory budget is exhausted. Let Kafka handle the queue — that's what it's for.

03
Memory budgets need dual limits

"Max 512 messages" isn't enough — one 100 MiB message blows the budget. Track both message count AND byte size together.

04
Timeouts must be per-operation

One global 60-second timeout for (cache + ES + MongoDB) wastes latency budget and hides which operation is actually slow. Separate timeouts let each operation fail independently and surface the real bottleneck.

05
Logging in hot paths kills performance

1,000 Info logs/sec = 1,000 heap allocations/sec = GC pressure. Use Debug for cache hits; Info for misses and errors only. Structured logging fields still allocate.

06
Batch processing reduces peak memory by 98%

Loading 50K Redis keys at once → 480 MiB spike. Processing 100 at a time with SCAN → ~10 MiB steady state. Always scan-and-batch for bulk operations — never KEYS *.

07
Cache your Redis Lua scripts

500 SCRIPT LOAD calls/sec = connection pool exhaustion + wasted bandwidth. Pre-compute the SHA1 at startup with SCRIPT LOAD once, then call EVALSHA for every subsequent execution.
